# ELF Inspection: firmware.elf

This document explains the `arm-none-eabi-objdump -d firmware.elf` output for `STM32-NUCLEO-F446RE/PWM/build/firmware.elf`.

## 1. OBJ Dump command

Use the ARM-specific disassembler:
```bash
arm-none-eabi-objdump -d build/firmware.elf
```

The output is a disassembly of the `.text` and related code sections from the ELF executable.

---

## 2. File format

The ELF is:
- `ELF32` (32-bit)
- little-endian
- architecture `ARM`
- type `EXEC` (executable)
- EABI version 5
- soft-float ABI

This matches the STM32 ARM Cortex target used by the toolchain.

---

## 3. Section `.text`

The disassembly shows:
- `__isr_vector` at `0x08000000`
- `main` at `0x08000040`
- `Reset_Handler` at `0x08000090`
- exception handlers at `0x0800009a` through `0x080000aa`
- literal pool entries embedded after `main`

### 3.1 `__isr_vector`

This is the interrupt vector table placed at the beginning of Flash.
Each `.word` is an address:
- `0x08000000` or `0x00001000` is the initial stack pointer value
- following words are handler addresses for:
  - `Reset_Handler`
  - `NMI_Handler`
  - `HardFault_Handler`
  - `MemManage_Handler`
  - `BusFault_Handler`
  - `UsageFault_Handler`
  - `SVC_Handler`
  - `DebugMon_Handler`
  - `PendSV_Handler`
  - `SysTick_Handler`

This is the standard Cortex-M vector table layout.

### 3.2 `main`

At `0x08000040`, the compiled `main()` function begins.

#### Register and address loading
```asm
8000040: 4910      ldr r1, [pc, #64]   @ loads 0x40023000
8000042: 4a11      ldr r2, [pc, #68]   @ loads 0x40020000
```
- These are PC-relative loads from the literal pool at the end of `main`.
- `r1` becomes `0x40023000`, a base address for RCC registers.
- `r2` becomes `0x40020000`, the GPIOA base address.

#### Enable GPIOA clock
```asm
8000044: f8d1 3830   ldr.w r3, [r1, #2096]    @ read RCC_AHB1ENR
8000048: f043 0301   orr.w r3, r3, #1        @ set bit 0
800004c: f8c1 3830   str.w r3, [r1, #2096]   @ write RCC_AHB1ENR
```
- `r1 + 0x830` = `0x40023830`.
- This sets `RCC_AHB1ENR |= 1`, enabling GPIOA clock.

#### Configure PA5 as output
```asm
8000050: 6813      ldr r3, [r2, #0]
8000052: f423 6340 bic.w r3, r3, #3072   @ clear bits 10-11
8000056: 6013      str r3, [r2, #0]
8000058: 6813      ldr r3, [r2, #0]
800005a: f443 6380 orr.w r3, r3, #1024   @ set bit 10
800005e: 6013      str r3, [r2, #0]
```
- `r2 + 0x0` = `GPIOA_MODER`.
- `bic` clears bits `0x0C00`; `orr` sets `0x0400`.
- This configures PA5 as general-purpose output.

#### Toggle the LED loop
```asm
8000060: 6951      ldr r1, [r2, #20]
8000062: 4b0a      ldr r3, [pc, #40]       @ loads 0x0007a120
8000064: f041 0120 orr.w r1, r1, #32        @ set PA5 bit
8000068: 6151      str r1, [r2, #20]
800006a: bf00      nop
800006c: 3b01      subs r3, #1
800006e: d1fc      bne.n 800006a

8000070: 6951      ldr r1, [r2, #20]
8000072: 4b06      ldr r3, [pc, #24]
8000074: f021 0120 bic.w r1, r1, #32        @ clear PA5 bit
8000078: 6151      str r1, [r2, #20]
800007a: bf00      nop
800007c: 3b01      subs r3, #1
800007e: d1fc      bne.n 800007a
8000080: e7ee      b.n 8000060
```
- `r2 + 20` = `GPIOA_ODR`.
- `orr.w` sets bit 5 to turn LED on.
- `bic.w` clears bit 5 to turn LED off.
- The delay is implemented with a counted loop using `subs` and `bne`.
- `e7ee` loops back to the on/off sequence forever.

#### Literal pool values
```asm
8000084: 40023000    .word 0x40023000
8000088: 40020000    .word 0x40020000
800008c: 0007a120    .word 0x0007a120
```
- These are the addresses and delay constant used by `main()`.
- `0x0007a120` equals decimal `500000`.

### 3.3 `Reset_Handler`

At `0x08000090`:
```asm
8000090: f8df d018 ldr.w sp, [pc, #24] @ 0x00001000
8000094: f7ff ffd4 bl 8000040 <main>
8000098: e7fe b.n 8000098 <Reset_Handler+0x8>
```
- The initial stack pointer is loaded from the literal pool at `0x080000ac`.
- `sp` becomes `0x00001000`.
- Then `main()` is called.
- After `main()` returns, it loops forever.

### 3.4 Exception handlers

The handlers at `0x0800009a` through `0x080000aa` are simple infinite loops:
```asm
e7fe b.n <self>
```
- This is typical for weak default handlers in embedded firmware.
- It stops execution if the exception is triggered.

### 3.5 `.init` and `.fini`

The ELF also contains very small stubs for `_init` and `_fini`:
- `_init` pushes registers then does `nop`
- `_fini` is the same
- These are standard placeholders generated by the toolchain and are usually unused in minimal firmware

## 4. What this output means

### Vector table and startup
- The first bytes of the ELF contain the MCU vector table, not normal code.
- The first vector is the initial stack pointer.
- The second vector is `Reset_Handler`.
- This is how Cortex-M boots from Flash.

### Code generation style
- The compiler uses PC-relative literal pool loads to reference constants and addresses.
- It uses wide Thumb instructions like `ldr.w`, `orr.w`, and `bic.w` for immediate values.
- The loop is implemented with `subs` + `bne`, matching the C `while` delay structure.

### Peripheral access
- Addresses are loaded from the literal pool and combined with offsets.
- This is equivalent to the high-level macros in `main.c`:
  - `RCC_AHB1ENR |= 0x00000001;`
  - `GPIOA_MODER &= ~(0x00000C00);`
  - `GPIOA_MODER |= 0x00000400;`
  - `GPIOA_ODR |= 0x00000020;`
  - `GPIOA_ODR &= ~0x00000020;`

---

## 5. Key takeaways
- `arm-none-eabi-objdump -d` shows the actual instructions the MCU will execute.
- The ELF includes both the interrupt vector table and compiled code.
- ARM Cortex-M firmware starts with the vector table at `0x08000000`.
- `main` uses a literal pool and page-aligned code to access hardware registers.
- `Reset_Handler` initializes `sp` and transfers control to `main`.
